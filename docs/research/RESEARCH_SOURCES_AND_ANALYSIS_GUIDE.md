## Broad Scan Analysis Instructions

### Phase 1: Repository Audit (Internal Analysis)

```markdown
## STEP 1: Code Inventory Scan
───────────────────────────────────────────────────────────────────

1.1 Run automated inventory:
    ```bash
    find . -type f -name "*.py" | wc -l  # Count Python files
    find . -type f -name "*.md" | wc -l  # Count documentation
    find . -type f -name "*.sv" | wc -l  # Count SystemVerilog
    ```

1.2 Identify all claims-making files:
    - Read CLAIMS_PRACTICING_FILES.txt
    - Cross-reference with LICENSE-CLAIMS-NC.md
    - Flag files with performance assertions

1.3 Extract all mathematical formulas:
    - Search for LaTeX equations in .md and .tex files
    - Catalog all transform definitions
    - List all complexity claims (O(N), O(N log N), etc.)
```

### Phase 2: Literature Comparison (External Validation)

```markdown
## STEP 2: Prior Art Search
───────────────────────────────────────────────────────────────────

2.1 For each claimed innovation, search:
    □ arXiv: "[technique name] transform"
    □ Google Scholar: "[technique name] signal processing"
    □ IEEE Xplore: "[technique name]"
    □ ACM DL: "[technique name]"

2.2 Document findings in format:
    | Claim | Prior Art Found | Year | Difference |
    |-------|-----------------|------|------------|
    | ...   | ...             | ...  | ...        |

2.3 Check patent databases:
    □ USPTO: https://patft.uspto.gov/
    □ Google Patents: https://patents.google.com/
    □ EPO: https://worldwide.espacenet.com/
```

### Phase 3: Benchmark Reproduction

```markdown
## STEP 3: Reproduce All Benchmarks
───────────────────────────────────────────────────────────────────

3.1 Run official benchmark suite:
    ```bash
    cd benchmarks
    python run_all_benchmarks.py --variants
    ```

3.2 Compare against reference implementations:
    □ FFT: scipy.fft vs RFT
    □ Compression: zstd/brotli vs RFTMW
    □ Quantum: Qiskit/Cirq vs QSC

3.3 Document deviation from claims:
    | Benchmark | Claimed | Measured | Delta |
    |-----------|---------|----------|-------|
    | ...       | ...     | ...      | ...   |
```

### Phase 4: Mathematical Verification

```markdown
## STEP 4: Prove/Disprove Theorems
───────────────────────────────────────────────────────────────────

4.1 For each theorem in docs/proofs/:
    □ Verify assumptions are stated
    □ Check proof steps are complete
    □ Run numerical validation tests

4.2 Test unitarity claims:
    ```python
    from algorithms.rft.core import CanonicalTrueRFT
    rft = CanonicalTrueRFT(1024, validate_unitarity=True)
    # Should not raise ValueError
    ```

4.3 Verify frame bounds:
    □ Check Gram matrix condition number
    □ Verify Parseval equality holds
    □ Test reconstruction error < 1e-12
```

### Phase 5: Cross-Reference Standards

```markdown
## STEP 5: Standards Compliance Check
───────────────────────────────────────────────────────────────────

5.1 Cryptography:
    □ Compare against NIST PQC requirements
    □ Check key sizes meet security levels
    □ Verify avalanche effect claims

5.2 Medical:
    □ Check FDA/CE compliance statements
    □ Verify "research only" disclaimers present
    □ Cross-reference with IEC 62304 (medical software)

5.3 Compression:
    □ Compare against standard codecs (H.264, JPEG)
    □ Verify bits-per-pixel calculations
    □ Check against Shannon limits
```

## Domain-Specific Research Protocols

### Protocol A: Transform Analysis

```markdown
OBJECTIVE: Validate RFT mathematical properties

SOURCES TO CHECK:
1. arXiv math.NA - Numerical analysis papers on unitary transforms
2. IEEE Signal Processing - Transform coding papers
3. Britannica - Fourier analysis fundamentals

VALIDATION STEPS:
□ Confirm basis orthonormality: Ψ†Ψ = I
□ Test energy preservation: ||x||² = ||Ψx||²
□ Verify O(N log N) complexity via timing
□ Compare sparsity vs FFT on test signals

EXPECTED OUTCOME:
- Unitarity error < 1e-12
- Timing scales as N log N
- Sparsity advantage on φ-structured signals only
```

### Protocol B: Quantum Simulation

```markdown
OBJECTIVE: Validate quantum symbolic engine claims

SOURCES TO CHECK:
1. arXiv quant-ph - Quantum simulation papers
2. Qiskit/Cirq documentation
3. Nielsen & Chuang textbook

VALIDATION STEPS:
□ Compare fidelity vs Qiskit for Bell state
□ Measure gate application time
□ Test scaling: O(N) vs O(2^N) claimed

EXPECTED OUTCOME:
- Fidelity ≥ 0.9999 for standard circuits
- Correct scaling for symbolic representation
- Clear limitation disclosure for large circuits
```

### Protocol C: Cryptographic Security

```markdown
OBJECTIVE: Validate post-quantum security claims

SOURCES TO CHECK:
1. IACR ePrint - Lattice cryptography papers
2. NIST PQC documentation
3. liboqs reference implementations

VALIDATION STEPS:
□ Run NIST statistical test suite (STS)
□ Measure avalanche effect (target: 50%)
□ Check key/ciphertext sizes vs standards
□ Verify no known attacks apply

EXPECTED OUTCOME:
- STS pass rate > 95%
- Avalanche 49-51%
- Security level clearly stated
```

### Protocol D: Medical Signal Processing

```markdown
OBJECTIVE: Validate ECG/EEG processing claims

SOURCES TO CHECK:
1. PhysioNet database documentation
2. medRxiv - Signal processing in medicine
3. FDA guidance on medical device software

VALIDATION STEPS:
□ Test on MIT-BIH arrhythmia database
□ Compare PSNR vs published baselines
□ Verify "research only" disclaimers

EXPECTED OUTCOME:
- Roundtrip error < 1e-14
- PSNR improvement stated with confidence intervals
- Clear non-clinical-use disclaimer
```
